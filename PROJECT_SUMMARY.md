## 1. What This App Does

This is a **Shopify embedded app** for subscription product rotation, designed to work with **Loop Subscriptions**.

**The core idea:**
- A merchant sets up a "Rotation Group" for a subscription product (the *target product*)
- They add an ordered list of *rotation products* (the replacement sequence)
- When a **Loop renewal order** arrives, the app automatically swaps the original subscription product for the next product in the rotation sequence
- Prices are matched using Shopify's Order Edit API (discounts are applied so the customer always pays what they originally subscribed to)
- After the last rotation product, the sequence cycles back to position 1

**Example:**
> Customer subscribes to "100,000 Whys (Copy)" for $43.95/month.
> Rotation sequence: [The Growing Up Guide For Teens → The Magic Physics Lab → ...]
> Month 1: original product is kept, rotation instance is created.
> Month 2: order is edited to swap in "The Growing Up Guide For Teens" at $43.95.
> Month 3: swaps in "The Magic Physics Lab" at $43.95. And so on.

## 4. Database Schema

### `Session`
Standard Shopify session storage. Managed automatically by `@shopify/shopify-app-session-storage-prisma`. Do not modify.

### `ShopSetting`
One row per shop. Created on first webhook. Acts as the parent for `RotationGroup` (foreign key constraint).

```
shop (unique) | isActive | createdAt | updatedAt
```

### `RotationGroup`
One row per target subscription product per shop.

```
id (cuid)
shop
targetProductId    — GID: "gid://shopify/Product/XXXXX"
targetProductTitle — display name
isActive           — when false, rotation is paused for this group
@@unique([shop, targetProductId])
```

### `RotationItem`
The ordered list of replacement products for a group. Each item = one product that the subscription will be swapped TO.

```
id (cuid)
rotationGroupId
productId     — GID
variantId     — GID of the default/fallback variant
productTitle
variantTitle  — e.g. "Teen Boys", "Default Title"
sortOrder     — 0-based integer; determines rotation order
isActive      — when false, item is skipped during rotation
imageUrl      — cached product thumbnail
price         — cached variant price (display only)
```

### `SubscriptionInstance`
One row per customer subscription. Tracks WHERE in the rotation sequence that customer currently is.

```
id (cuid)
shop
customerId             — Shopify customer numeric ID (string)
originalOrderId        — ID of the first order that created this instance
subscriptionContractId — Loop contract GID (nullable)
targetProductId        — GID of the product being rotated
currentIndex           — integer: index into rotationItems array
uniqueKey              — "{shop}:{customerId}:{numericProductId}:{originalOrderId}"
status                 — "ACTIVE" | "PAUSED" | "CANCELLED"
rotationGroupId
```

**uniqueKey** prevents duplicate instances. It incorporates the original order ID so each new subscription purchase gets a fresh instance even for the same customer/product.

### `RotationLog`
Audit trail. One row per rotation attempt (SUCCESS, FAILED, or SKIPPED).

```
id (cuid)
shop
orderId              — GID of the renewal order
customerId
targetProductTitle   — what was originally in the order
rotationProductTitle — what was swapped in
status               — "SUCCESS" | "FAILED" | "SKIPPED"
message              — error message for FAILED, skip reason for SKIPPED
createdAt
```

**Auto-pruning:** rows older than 7 days are deleted after every webhook run. Maximum 50 rows per shop are kept.

## 5. Core Rotation Logic

**Entry point:** `app/services/rotation.server.js` → `processOrderWebhook(shop, order, admin)`

### Flow on `orders/create` webhook

```
1. Upsert ShopSetting for the shop.

2. Determine if this is a Loop renewal:
   order.source_name === "subscription_contract_checkout_one"

3. For each active RotationGroup:
   a. Find line items in the order matching targetProductId.
   b. If no matching line items → skip this group.

4. NEW SUBSCRIPTION PURCHASE (not a renewal):
   - Cancel any stale ACTIVE instances for this customer+product
     (catches Loop cancellations that didn't send our webhook).
   - Create a fresh SubscriptionInstance with currentIndex=0.
   - Do NOT edit the order (first order keeps the original product).

5. RENEWAL ORDER:
   a. Find existing SubscriptionInstance via contractId or customerId+product.
   b. If none exists → create one (handles renewal arriving before first order).
   c. Idempotency check: if a RotationLog already exists for this order+customer → skip.
   d. Call rotateOrderItems().

6. After all groups processed → pruneOldLogs() (non-blocking).
```

### `rotateOrderItems()`

```
1. Get activeItems = group.rotationItems.filter(isActive), sorted by sortOrder.
2. If empty → log SKIPPED "No active rotation items".
3. nextIndex = instance.currentIndex % activeItems.length
4. nextItem = activeItems[nextIndex]
5. newIndex = (nextIndex + 1) % activeItems.length

6. SELF-ROTATION GUARD:
   If nextItem.productId === group.targetProductId:
   → advance index, log SKIPPED "Rotation item is the target product", return.

7. Call performOrderEdit() (see Section 6).
8. Update instance.currentIndex = newIndex.
9. Log SUCCESS.

10. ERROR HANDLING:
    - err.concurrent = true → silent return (another webhook run got there first).
    - Any other error → log FAILED, rethrow.
```

### `extractContractId(order)`
Scans `order.note_attributes[]` and all line item `properties[]` for a value matching the pattern `gid://shopify/SubscriptionContract/\d+`. This is how Loop embeds the subscription contract GID in the order.

## 6. Order Edit Flow

**File:** `app/services/order-edit.server.js` → `performOrderEdit({ admin, orderGid, targetLineItems, nextItem, currency })`

### Step 1: Determine Case 1 vs Case 2

- Fetch all variants of the *rotation product* (`nextItem.productId`).
- Build a map: `variantTitle → variant`.
- Check if ALL subscription line items' variant titles exist in the map.
  - **Case 2 (variant match):** Every subscription line item has a matching variant in the rotation product. Each variant is swapped individually (e.g., "Teen Boys" → "Teen Boys" of new product).
  - **Case 1 (default variant):** Use the rotation item's stored default variant for all line items.

### Step 2: Begin Order Edit
`orderEditBegin(orderId)` → returns `calcOrderId` and `calcLineItems[]` (the calculated/editable versions of line items).

### Step 3: Zero out target line items
For each subscription line item → `orderEditSetQuantity(calcLineItemId, 0)`.

**Concurrent webhook guard:** If the mutation returns "cannot be edited because it is removed" → another webhook run already zeroed this item → throw `err.concurrent = true`.

### Step 4: Add rotation items with exact pricing

**Critical: Shopify's `orderEditAddLineItemDiscount` `fixedValue` is applied PER UNIT, not per line total.**

**The fractional cent problem:** If a line total is not evenly divisible by quantity (e.g., $56.21 / 2 = $28.105), there is no valid 2-decimal representation for the per-unit price. Applying a single `qty=2` add with a rounded discount will always produce a 1-cent refund.

**Fix — `addUnitsExact()` inner function:**
```
For each unit in targetLineItem.quantity:
  totalCents    = Math.round(lineTotal * 100)          // integer cents
  baseUnitCents = Math.floor(totalCents / qty)         // floor per unit
  unitCents     = last unit? (totalCents - baseUnitCents * i) : baseUnitCents
  discountCents = variantCents - unitCents
  → add qty=1 line item, apply per-unit discount
```
The last unit absorbs the remainder (±1 cent). No floating-point drift.

**`lineTotal(li)` helper — reliable for Loop orders:**
Loop subscription orders set `final_line_price = "0.00"` on renewal. So:
1. Try `final_line_price` (if > 0 use it).
2. Fall back to `price × quantity - sum(discount_allocations)`.
3. `discount_allocations` is available on all Shopify plans; `total_discount` is Shopify Plus only.

### Step 5: Commit
`orderEditCommit()` — if "Could not save the order edit" → another webhook run committed first → throw `err.concurrent = true`.

## 8. Webhook Handlers

### `webhooks.orders.create.jsx` ← most important
Triggers rotation. Calls `processOrderWebhook(shop, order, admin)`.

HMAC signature is verified automatically by the Shopify middleware (`authenticate.webhook`). The webhook is registered in `shopify.app.toml`.

**Duplicate webhook protection (3 layers):**
1. `RotationLog` idempotency check: if order+customer already logged → skip.
2. "already removed" error on `orderEditSetQuantity` → `err.concurrent = true`.
3. "Could not save the order edit" on commit → `err.concurrent = true`.

### `webhooks.subscription-contracts.activate.jsx`
Re-activates `SubscriptionInstance` when a Loop contract is reactivated.

### `webhooks.subscription-contracts.cancel.jsx`
Sets `SubscriptionInstance.status = "CANCELLED"` when a Loop contract is cancelled.

### `webhooks.subscription-contracts.update.jsx`
Handles contract updates (e.g., product changes from Loop).

### `webhooks.app.uninstalled.jsx`
Cleans up shop data on app uninstall.

### `webhooks.app.scopes_update.jsx`
Handles scope changes.

## 10. Key Technical Decisions

### Why `source_name === "subscription_contract_checkout_one"` identifies Loop renewals
Loop Subscriptions sets this specific `source_name` value on all recurring renewal orders. First purchases use `"web"` or other values. This is the only reliable way to distinguish renewal from new subscription without querying the Subscription Contracts API.

### Why we don't use JSON line item snapshots
Early design stored the original line items as JSON blobs in `SubscriptionInstance`. Removed because:
- The live `order.line_items` from the webhook payload is always fresh and accurate
- Fewer fields = simpler schema = no risk of stale snapshot data

### Why `uniqueKey` includes `originalOrderId`
If a customer cancels and resubscribes to the same product, we want a NEW rotation sequence starting from position 0, not picking up where the old one left off. Including `originalOrderId` in the key guarantees each subscription purchase gets an independent instance.

### Why `expiringOfflineAccessTokens: false` in `shopify.server.js`
Disabled in development to prevent intermittent re-auth / login popups during development. Re-enable before App Store submission if token rotation is required by Shopify's review process.

### Why per-unit `qty=1` adds instead of one multi-unit add
Shopify's `orderEditAddLineItemDiscount` with `fixedValue` applies the discount amount PER UNIT. Adding a `qty=2` line item with a single discount of X means the actual line discount is `2X`. This caused incorrect prices (2× the intended discount). The fix: add one `qty=1` line item per unit, apply per-unit discount to each individually.

### Exit-iframe for OAuth redirects
In Shopify embedded apps, OAuth requires loading `accounts.shopify.com` which has `X-Frame-Options: DENY`. When the session expires and the app needs to re-auth:
- `login()` throws a `redirect()` response
- React Router would follow this in the iframe → "refused to connect"
- **Fix:** `auth.login/route.jsx` catches the thrown Response, returns the OAuth URL as JSON data, and the client uses `window.top.location.assign(url)` to exit the iframe before OAuth
- Also covered by `ExitIframeRedirect` component in `app.jsx` `ErrorBoundary`

### `{shop: null}` logs are normal
The `[shopify-app/INFO] Authenticating admin request | {shop: null}` log appears on every navigation because client-side React Router routes don't include `?shop=` in the URL. Authentication succeeds via the App Bridge JWT session token (in the `Authorization` header), not the URL param. This is expected behavior — not a bug.

## 11. Known Bugs Fixed & Why

| Bug | Symptom | Root cause | Fix |
|---|---|---|---|
| Self-rotation | Order shows original product unchanged | Rotation item `productId === targetProductId` (user accidentally added the subscription product itself to the rotation) | Skip + log SKIPPED + advance index |
| Wrong price (b24.52 instead of b56.21) | Discount double-applied | `fixedValue` in `orderEditAddLineItemDiscount` is per unit, not per line | Calculate discount per unit, add qty=1 individually |
| 1-cent refund (b56.20 instead of b56.21) | Shopify creates a tiny refund | `56.21 / 2 = 28.105` can't be represented in 2 decimal places | `addUnitsExact`: integer-cent math, last unit absorbs remainder |
| "Could not save the order edit" | Terminal error on every other renewal | Loop fires `orders/create` twice; second commit fails because first already committed | `err.concurrent = true` flag; silently skip in `rotateOrderItems` |
| "cannot be edited because it is removed" | Terminal error | Two concurrent webhook runs; second run tries to zero already-removed items | `setLineItemQuantity` returns `false` → `err.concurrent = true` |
| "accounts.shopify.com refused to connect" | App breaks, shows connection error | OAuth redirect followed inside iframe | Exit-iframe via `window.top.location.assign` in `auth.login/route.jsx` |
| Aside cards misaligned | Right-side info cards appear at top | `<Breadcrumbs>` div placed as direct `s-page` child broke grid layout | Move breadcrumbs outside `<s-page>` using React Fragment |

## 15. Production Checklist

Before going live:

- [ ] Set `expiringOfflineAccessTokens: true` in `shopify.server.js`
- [ ] Switch `DATABASE_URL` to a production Postgres database
- [ ] Run `npx prisma migrate deploy` on production DB
- [ ] Confirm all webhook endpoints are registered and receiving events
- [ ] Test a full rotation cycle: subscribe → renewal → product swap → check for refund
- [ ] Test with qty > 1 line items (the fractional-cent edge case)
- [ ] Test concurrent webhook delivery (Loop often fires twice; both should succeed without duplicate edits)
- [ ] Verify the `orders/create` webhook is live in Shopify Partner Dashboard
- [ ] Confirm `SHOPIFY_APP_URL` points to production host (not ngrok)
- [ ] Consider raising the RotationLog cap from 50 rows / 7 days if more history is needed
- [ ] Test the exit-iframe OAuth flow in production (session expiry → re-auth should not break the embedded app)
